# Architecture Open Questions

This register preserves contradictions and specification gaps from the authoritative sources. Recommendations remain proposals until accepted by an implementation decision. Q-003 is retained for historical traceability but is resolved and superseded by [ADR 0001](../decisions/0001-phase-1-transport-lifecycle.md).

| ID | Decision needed | Conflicting evidence | Implementation impact | Recommended resolution | Owning phase |
|---|---|---|---|---|---|
| `Q-001` | Canonical universal editor-script tool name | `00-master-architecture-and-standards.md` §1 names Tier B `run_editor_script`, while `phase-02-introspection-and-universal-primitive.md` §§2 and 10 expose and deliver `godot_script_run`. The master §6 naming standard also gives `godot_script_run` as its example, while Phase 2 §3 still calls the produced interface the `run_editor_script` contract. | The choice fixes the registry key, schemas, safety rules, audit records, tests, examples, and client discovery. Registering both would duplicate a dangerous execution surface unless alias semantics were designed. | **Proposal (not source truth):** Use `godot_script_run` as the sole public MCP tool name and reserve `run_editor_script` for the conceptual capability or internal contract. Add no alias without a separate compatibility decision. | Phase 2 |
| `Q-002` | Whether Phase 2 is a hard prerequisite for Phases 4–6 | `00-master-architecture-and-standards.md` §7 and the Phase 2 preamble call Phase 2 the spine and say Phases 3–6 proceed once it lands. In contrast, `phase-04-code-intelligence-lsp.md` §3 and `phase-06-batch-filesystem-and-assets.md` §3 consume only Phase 1 and define isolated development, while `phase-05-runtime-and-debug.md` §3 explicitly consumes Phase 2 execution. | A blanket gate unnecessarily serializes LSP and batch work; removing it universally breaks runtime-hook injection. The decision changes the dependency graph, stubs, CI gates, scheduling, and phase-completion criteria. | **Proposal (not source truth):** Make Phase 1 the hard prerequisite for Phases 4 and 6. Permit Phase 5 ProcessRunner/DAP work after Phase 1, but require Phase 2 execution for full Phase 5 completion and runtime-bridge injection. Treat the broader spine language as a coordination milestone. | Architecture/Phases 4–6 |
| `Q-003` — **Resolved / superseded by [ADR 0001](../decisions/0001-phase-1-transport-lifecycle.md)** | Heartbeat transport and relationship to `core.ping` | `00-master-architecture-and-standards.md` §6 specifies a JSON-RPC `ping`/`pong` heartbeat and says the plugin reconnects. `phase-01-foundation-and-transport.md` §§5–6 instead make the plugin the WebSocket server and give the TypeScript `ws-client` heartbeat and reconnect ownership; §§6 and 8 separately define `core.ping`, `godot_ping`, and heartbeat acceptance. | Historical conflict: the frame type, initiator, timeout owner, reconnect owner, and meaning of the probe test were undefined, allowing native control frames, JSON-RPC heartbeat calls, or duplicate timers. | **Accepted resolution:** The TypeScript client owns one JSON-RPC `core.ping` heartbeat and reconnects with delays from 1 second through a 60-second cap. `godot_ping` remains the public end-to-end probe. | Phase 1 |
| `Q-004` | Local editor authentication, TLS, and multi-client policy | `00-master-architecture-and-standards.md` §§2–3 calls the endpoint local and labels it `localhost`, while `phase-01-foundation-and-transport.md` §4 says “Two processes, one link” but §6 says the server accepts clients. `phase-07-hardening-safety-concurrency-observability.md` §5 requires one enforcement point and one mutation lane, yet no source defines authentication, TLS, binding, client selection, or takeover. | An unspecified listener could admit other local or network clients to arbitrary execution, bypass server-side safety/audit, or create competing mutation queues. Later security changes would alter setup, configuration, handshake, errors, and reconnect tests. | **Proposal (not source truth):** Enforce IPv4/IPv6 loopback, authenticate before dispatch with a high-entropy per-project or per-install secret, and allow one active TypeScript control-plane connection. Permit plaintext `ws://` only on enforced loopback; any future remote mode must explicitly require authenticated `wss://`. | Phase 7 |
| `Q-005` — **Resolved / accepted in Phase 3** | Undo semantics for project-setting mutation | `phase-03-curated-editor-mutation-tier.md` §§1 and 5 promise every Tier A mutation is one Ctrl-Z away and disqualify operations without do/undo. Its §6 nevertheless permits `godot_project_setting_set` to run non-undoably when marked destructive, and §9 only suggests snapshot/restore. | The implementation snapshots prior presence and value, preflights restoration and persistence, and blocks subsequent writes if recovery cannot be proven. | **Accepted resolution:** Execute the Tier A tool only when one UndoRedo action restores the exact previous state, including removal of a previously absent key; otherwise reject and fail closed. | Phase 3 |
| `Q-006` | Safety classification for arbitrary headless GDScript | `phase-06-batch-filesystem-and-assets.md` §§2, 5, and 7 accept arbitrary source in `godot_headless_run_script` but assign it no safety or mutation classification. The master §6 and `phase-07-hardening-safety-concurrency-observability.md` §§2, 5, and 7 explicitly gate and queue `run_editor_script`, while Phase 7 also claims uniform safety across every tool. | Without a classification, arbitrary headless code may evade read-only blocking, confirmation, execution guards, and mutation serialization. The decision controls MCP annotations, queue routing, limits, deny-list behavior, and mode-conformance tests. | **Proposal (not source truth):** Classify it conservatively as `readOnlyHint:false`, `destructiveHint:true`, `idempotentHint:false`, `openWorldHint:true`, and mutating. Block it in `read_only`, require confirmation in `confirm_destructive`, apply process/exec guards, and use the project mutation lane. | Phase 7 |
| `Q-007` | Whether rejected calls always reach audit logging | `phase-07-hardening-safety-concurrency-observability.md` §4 draws audit after the handler while the blocked branch exits directly from the policy gate. Its §5 simultaneously requires an append-only record of every tool call, and Phase 8 §9 requires audit active on every tool. | Pipeline placement determines whether policy rejections and pre-handler failures are visible for trust and forensics. An after-handler-only stage cannot record every outcome and risks duplicate or missing entries when moved. | **Proposal (not source truth):** Create an audit context at registry entry and finalize exactly one redacted record in an outer `finally` path for success, invalid arguments, policy rejection, queue timeout, handler failure, and post-dispatch transport loss. Log unknown requests separately at the transport layer. | Phase 7 |
| `Q-008` | Read consistency during an in-progress mutation | The master §6 and Phase 7 §§5 and 8 say reads run concurrently and must proceed during a long mutation. Phase 7 §§4–5 invalidates caches only after the mutation and claims this avoids staleness, leaving reads able to see cached pre-state, partial live state, or repopulate stale cache after invalidation. | The missing consistency contract affects queue/cache coordination, observable state, failed partial mutations, cache-tag generations, and deterministic concurrency tests. Arbitrary scripts make affected tags unknowable. | **Proposal (not source truth):** Fence affected state domains at mutation start, invalidate at start and in `finally`, and discard read results if their generation changed before cache insertion. Let unrelated reads continue; make arbitrary-script mutations use a global fence and invalidation. | Phase 7 |
| `Q-009` | Allowed roots for export output paths | The master §6 says all filesystem paths stay under the project root, and `phase-06-batch-filesystem-and-assets.md` §§5–7 make `FsGuard` root-jail filesystem and asset-write paths. Phase 6 §8 nevertheless requires an export to a temp directory without saying that directory is inside the project. | The answer controls export validation, CI temp output, absolute paths, directory creation, overwrite confirmation, symlink handling, and whether Phase 7 blocks legitimate builds. | **Proposal (not source truth):** Keep direct filesystem/asset writes project-root-only. Allow exports under the project root, a tool-owned session temp/export directory, or canonical user-configured export roots; canonicalize paths, reject escapes, and treat overwrite as destructive. | Phases 6–7 |
| `Q-010` | DAP versus ProcessRunner launch ownership | `phase-05-runtime-and-debug.md` §4 says ProcessRunner spawns the debug process while DAP supports both `launch` and `attach`. Its §6 first says DAP attaches to an already launched session, then says to “launch under DAP”; §2 exposes both `godot_run_project` and `godot_debug_launch`. | Dual ownership can double-spawn, split PID/stdout ownership, race stop/teardown, use inconsistent launch arguments, or attach DAP to the wrong process. | **Proposal (not source truth):** Introduce one runtime-session coordinator. By default ProcessRunner alone owns OS spawn, PID, output, and stop; DAP performs protocol initialization plus attach or the required launch handshake without independently spawning. Both public tools delegate to that coordinator. | Phase 5 |
| `Q-011` | Host resolution of Godot's `user://` IPC path | `phase-05-runtime-and-debug.md` §§4 and 7 say the host writes, reads, and deletes `user://.mcp` files, but `user://` is a Godot virtual path. The screenshot sketch returns both logical `path` and `absPath`, while the master §6 configuration defines no user-data-root discovery contract or owner for producing the absolute path. | File IPC, screenshots, cleanup, path enforcement, and concurrent-project isolation cannot work safely if the host guesses platform- and project-specific Godot data directories. | **Proposal (not source truth):** Have Godot resolve the IPC root and publish its canonical absolute path plus a random session ID over a non-file bootstrap channel. The host registers only that session subdirectory as allowed and never derives `user://` from OS conventions. | Phase 5 |
| `Q-012` | Local-socket negotiation and file-IPC fallback | `phase-05-runtime-and-debug.md` §§5 and 9 says to prefer a local socket and fall back to file IPC, but §6 implements only sequenced files and defines no negotiation. `00-competitive-research.md` likewise says “where feasible” without specifying socket type, endpoint, authentication, timeout, fallback trigger, or switching behavior. | Independent assumptions can strand peers on different transports, expose a listener, replay a state-changing request during fallback, and make startup or teardown nondeterministic. | **Proposal (not source truth):** Negotiate a versioned, authenticated loopback endpoint with session ID, token, capabilities, and bounded pre-request handshake. If setup fails, lock both sides to sequenced file IPC for that session and record why; never switch or replay after a request may have executed. | Phase 5 |
| `Q-013` | Canonical Node.js product engine requirement | `phase-08-packaging-resources-prompts-eval-and-production.md` §4 requires an `engines` pin, but §6 states “Node ≥18/20.” `00-master-architecture-and-standards.md` §4 selects TypeScript and `npx` without choosing a Node runtime floor. | The ambiguity blocks a definitive `package.json` engine, minimum-runtime CI job, dependency/toolchain baseline, cold-`npx` verification, and install documentation. | **Proposal (not source truth):** Set `engines.node` to `>=20` and test Node 20 as the minimum because it is the stricter source-listed alternative. | Phase 8 |
| `Q-014` | Canonical `add-feature-to-scene` prompt name | `phase-08-packaging-resources-prompts-eval-and-production.md` §2 names the public prompt `add-feature-to-scene`, while §6 calls the implementation `add-feature`; no alias or preferred identifier is declared. | The public identifier affects prompt registration and discovery, tests, documentation, evaluation fixtures, and client-authored calls. | **Proposal (not source truth):** Use `add-feature-to-scene` as the sole canonical prompt identifier because the formal scope defines the public prompt surface with that name. | Phase 8 |
| `Q-015` | Supported Godot minor-version matrix | The master opening and §5 target Godot 4.6, require per-minor pinning, and call for tests on each supported minor. Phase 8 §§6, 8, and 9 refer to “any other pinned minors,” “target minor(s),” and “supported minor(s)” without naming any version beyond 4.6. | The unspecified set leaves CI dimensions, compatibility docs, plugin release claims, compatibility-shim coverage, and support policy indeterminate. | **Proposal (not source truth):** Make the initial supported matrix exactly Godot 4.6 and add a minor only after its full suite and compatibility row are green. | Phase 8 |
| `Q-016` | Reconnect acceptance-window duration | The master §6 and Phase 1 §6 define exponential retry delays from 1 to 60 seconds. Phase 1 §8 and Phase 8 §9 require reconnection “within backoff window” but define neither an end-to-end duration nor the timing origin; the 60-second value is an interval cap, not an acceptance window. | Release tests cannot set a deterministic deadline, and platforms can apply incompatible expectations about startup time versus retry time. | **Proposal (not source truth):** Require connection within 65 seconds after the plugin WebSocket listener becomes available: the 60-second maximum retry delay plus a 5-second handshake/test margin. Test the retry schedule separately. | Phase 8 |
