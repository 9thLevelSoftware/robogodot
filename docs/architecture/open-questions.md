# Architecture Open Questions

This register preserves contradictions and specification gaps from the authoritative sources. Recommendations remain proposals until accepted by an implementation decision. Q-003 is retained for historical traceability but is resolved and superseded by [ADR 0001](../decisions/0001-phase-1-transport-lifecycle.md).

| ID | Decision needed | Conflicting evidence | Implementation impact | Recommended resolution | Owning phase |
|---|---|---|---|---|---|
| `Q-001` — **Resolved / accepted in [ADR 0002](../decisions/0002-phase-2-execution-boundary.md)** | Canonical universal editor-script tool name | `00-master-architecture-and-standards.md` §1 names Tier B `run_editor_script`, while `phase-02-introspection-and-universal-primitive.md` §§2 and 10 expose and deliver `godot_script_run`. The master §6 naming standard also gives `godot_script_run` as its example, while Phase 2 §3 still calls the produced interface the `run_editor_script` contract. | The choice fixes the registry key, schemas, safety rules, audit records, tests, examples, and client discovery. Registering both would duplicate a dangerous execution surface unless alias semantics were designed. | **Accepted resolution:** Use `godot_script_run` as the sole public MCP tool name and reserve `run_editor_script` for the conceptual capability or internal contract. No alias is registered. | Phase 2 |
| `Q-002` — **Resolved for Phase 4 and Phase 6 / accepted in [ADR 0004](../decisions/0004-phase-6-batch-filesystem.md)** | Whether Phase 2 is a hard prerequisite for Phases 4–6 | `00-master-architecture-and-standards.md` §7 and the Phase 2 preamble call Phase 2 the spine. In contrast, the Phase 4 and Phase 6 dependency headers consume only Phase 1, while Phase 5 explicitly consumes Phase 2 execution. | A blanket gate unnecessarily serializes LSP and batch work; removing it universally breaks runtime-hook injection. | **Accepted Phase 4 resolution:** Phase 1 is the LSP API prerequisite; Phase 2 is coordination/regression only. **Accepted Phase 6 resolution:** Phase 1 config/logger/errors are the API prerequisite for FsGuard, headless, export, and UID tools. Headless uses an isolated process, not Phase 2 editor execution. Phase 2 remains a program coordination baseline. Phase 5 keeps its solid Phase 2 edge for runtime-bridge injection. | Architecture/Phases 4–6 resolved; Phase 5 unchanged |
| `Q-003` — **Resolved / superseded by [ADR 0001](../decisions/0001-phase-1-transport-lifecycle.md)** | Heartbeat transport and relationship to `core.ping` | `00-master-architecture-and-standards.md` §6 specifies a JSON-RPC `ping`/`pong` heartbeat and says the plugin reconnects. `phase-01-foundation-and-transport.md` §§5–6 instead make the plugin the WebSocket server and give the TypeScript `ws-client` heartbeat and reconnect ownership; §§6 and 8 separately define `core.ping`, `godot_ping`, and heartbeat acceptance. | Historical conflict: the frame type, initiator, timeout owner, reconnect owner, and meaning of the probe test were undefined, allowing native control frames, JSON-RPC heartbeat calls, or duplicate timers. | **Accepted resolution:** The TypeScript client owns one JSON-RPC `core.ping` heartbeat and reconnects with delays from 1 second through a 60-second cap. `godot_ping` remains the public end-to-end probe. | Phase 1 |
| `Q-004` — **Resolved / accepted in [ADR 0005](../decisions/0005-phase-7-hardening.md)** | Local editor authentication, TLS, and multi-client policy | `00-master-architecture-and-standards.md` §§2–3 calls the endpoint local and labels it `localhost`, while `phase-01-foundation-and-transport.md` §4 says “Two processes, one link” but §6 says the server accepts clients. `phase-07-hardening-safety-concurrency-observability.md` §5 requires one enforcement point and one mutation lane, yet no source defines authentication, TLS, binding, client selection, or takeover. | An unspecified listener could admit other local or network clients to arbitrary execution, bypass server-side safety/audit, or create competing mutation queues. Later security changes would alter setup, configuration, handshake, errors, and reconnect tests. | **Accepted resolution:** Loopback bind, high-entropy token auth before dispatch, and one authenticated control-plane client are normative (plugin implementation). Plaintext `ws://` only on loopback; remote `wss://` requires a future ADR. | Phase 7 resolved |
| `Q-005` — **Resolved / accepted in Phase 3** | Undo semantics for project-setting mutation | `phase-03-curated-editor-mutation-tier.md` §§1 and 5 promise every Tier A mutation is one Ctrl-Z away and disqualify operations without do/undo. Its §6 nevertheless permits `godot_project_setting_set` to run non-undoably when marked destructive, and §9 only suggests snapshot/restore. | The implementation snapshots prior presence and value, preflights restoration and persistence, and blocks subsequent writes if recovery cannot be proven. | **Accepted resolution:** Execute the Tier A tool only when one UndoRedo action restores the exact previous state, including removal of a previously absent key; otherwise reject and fail closed. | Phase 3 |
| `Q-006` — **Resolved / accepted in [ADR 0005](../decisions/0005-phase-7-hardening.md)** | Safety classification for arbitrary headless GDScript | `phase-06-batch-filesystem-and-assets.md` §§2, 5, and 7 accept arbitrary source in `godot_headless_run_script` but assign it no safety or mutation classification. The master §6 and `phase-07-hardening-safety-concurrency-observability.md` §§2, 5, and 7 explicitly gate and queue `run_editor_script`, while Phase 7 also claims uniform safety across every tool. | Without a classification, arbitrary headless code may evade read-only blocking, confirmation, execution guards, and mutation serialization. The decision controls MCP annotations, queue routing, limits, deny-list behavior, and mode-conformance tests. | **Accepted resolution:** `godot_headless_run` is mutating/destructive/openWorld; blocked in `read_only`; requires `confirmed: true` in `confirm_destructive`; enters the shared mutation lane. | Phase 7 resolved |
| `Q-007` — **Resolved / accepted in [ADR 0005](../decisions/0005-phase-7-hardening.md)** | Whether rejected calls always reach audit logging | `phase-07-hardening-safety-concurrency-observability.md` §4 draws audit after the handler while the blocked branch exits directly from the policy gate. Its §5 simultaneously requires an append-only record of every tool call, and Phase 8 §9 requires audit active on every tool. | Pipeline placement determines whether policy rejections and pre-handler failures are visible for trust and forensics. An after-handler-only stage cannot record every outcome and risks duplicate or missing entries when moved. | **Accepted resolution:** Registry creates audit context at entry and finalizes exactly one redacted record in `finally` for success, errors, and policy blocks. | Phase 7 resolved |
| `Q-008` — **Resolved / accepted in [ADR 0005](../decisions/0005-phase-7-hardening.md)** | Read consistency during an in-progress mutation | The master §6 and Phase 7 §§5 and 8 say reads run concurrently and must proceed during a long mutation. Phase 7 §§4–5 invalidates caches only after the mutation and claims this avoids staleness, leaving reads able to see cached pre-state, partial live state, or repopulate stale cache after invalidation. | The missing consistency contract affects queue/cache coordination, observable state, failed partial mutations, cache-tag generations, and deterministic concurrency tests. Arbitrary scripts make affected tags unknowable. | **Accepted resolution:** Mutations fence/invalidate at start and in `finally`; cache inserts are dropped if generation changed; openWorld/script/headless use a global fence. | Phase 7 resolved |
| `Q-009` — **Resolved / accepted in [ADR 0004](../decisions/0004-phase-6-batch-filesystem.md)** | Allowed roots for export output paths | The master §6 says all filesystem paths stay under the project root, and `phase-06-batch-filesystem-and-assets.md` §§5–7 make `FsGuard` root-jail filesystem and asset-write paths. Phase 6 §8 nevertheless requires an export to a temp directory without saying that directory is inside the project. | The answer controls export validation, CI temp output, absolute paths, directory creation, overwrite confirmation, symlink handling, and whether Phase 7 blocks legitimate builds. | **Accepted resolution:** Direct filesystem/asset writes remain project-root-only after realpath jail. Export outputs may target the project root, a server-owned session export directory under OS temp, or additional roots from `GODOT_MCP_EXPORT_ROOTS`. Overwrite requires explicit confirmation and is destructive. | Phase 6 resolved |
| `Q-010` — **Resolved / accepted in [ADR 0003](../decisions/0003-phase-5-runtime-session.md) and Phase 5** | DAP versus ProcessRunner launch ownership | `phase-05-runtime-and-debug.md` assigned contradictory launch and attach responsibilities. | Dual ownership could double-spawn, split PID/output ownership, and tear down the wrong process. | **Accepted resolution:** One runtime-session coordinator delegates every OS spawn, PID, output cursor, graceful/forced stop, and exact-child cleanup to ProcessRunner alone. DAP is attach-only, never spawns, and may degrade independently. Both public launch tools use this coordinator. | Phase 5 resolved |
| `Q-011` — **Resolved / accepted in [ADR 0003](../decisions/0003-phase-5-runtime-session.md) and Phase 5** | Host resolution of Godot's `user://` IPC path | The source required host access to a Godot virtual path without declaring discovery ownership. | Guessing platform user-data directories would break containment, cleanup, screenshots, and project isolation. | **Accepted resolution:** Godot resolves the canonical `user://` session root and publishes it through the authenticated plugin bootstrap. Node registers only the random `.mcp/<sessionId>` subdirectory, verifies canonical containment, and never derives `user://` from OS conventions. | Phase 5 resolved |
| `Q-012` — **Resolved / accepted in [ADR 0003](../decisions/0003-phase-5-runtime-session.md) and Phase 5** | Local-socket negotiation and file-IPC fallback | The source preferred a local socket but specified only file sequencing and no safe switching rule. | Peers could select different transports or replay a state-changing request during fallback. | **Accepted resolution:** Use a versioned, mutually authenticated loopback endpoint with bounded pre-request handshake. If it fails, both peers lock to sequenced file IPC before publication. Transport never switches and a published request is never replayed. | Phase 5 resolved |
| `Q-013` | Canonical Node.js product engine requirement | `phase-08-packaging-resources-prompts-eval-and-production.md` §4 requires an `engines` pin, but §6 states “Node ≥18/20.” `00-master-architecture-and-standards.md` §4 selects TypeScript and `npx` without choosing a Node runtime floor. | The ambiguity blocks a definitive `package.json` engine, minimum-runtime CI job, dependency/toolchain baseline, cold-`npx` verification, and install documentation. | **Proposal (not source truth):** Set `engines.node` to `>=20` and test Node 20 as the minimum because it is the stricter source-listed alternative. | Phase 8 |
| `Q-014` | Canonical `add-feature-to-scene` prompt name | `phase-08-packaging-resources-prompts-eval-and-production.md` §2 names the public prompt `add-feature-to-scene`, while §6 calls the implementation `add-feature`; no alias or preferred identifier is declared. | The public identifier affects prompt registration and discovery, tests, documentation, evaluation fixtures, and client-authored calls. | **Proposal (not source truth):** Use `add-feature-to-scene` as the sole canonical prompt identifier because the formal scope defines the public prompt surface with that name. | Phase 8 |
| `Q-015` | Supported Godot minor-version matrix | The master opening and §5 target Godot 4.6, require per-minor pinning, and call for tests on each supported minor. Phase 8 §§6, 8, and 9 refer to “any other pinned minors,” “target minor(s),” and “supported minor(s)” without naming any version beyond 4.6. | The unspecified set leaves CI dimensions, compatibility docs, plugin release claims, compatibility-shim coverage, and support policy indeterminate. | **Proposal (not source truth):** Make the initial supported matrix exactly Godot 4.6 and add a minor only after its full suite and compatibility row are green. | Phase 8 |
| `Q-016` | Reconnect acceptance-window duration | The master §6 and Phase 1 §6 define exponential retry delays from 1 to 60 seconds. Phase 1 §8 and Phase 8 §9 require reconnection “within backoff window” but define neither an end-to-end duration nor the timing origin; the 60-second value is an interval cap, not an acceptance window. | Release tests cannot set a deterministic deadline, and platforms can apply incompatible expectations about startup time versus retry time. | **Proposal (not source truth):** Require connection within 65 seconds after the plugin WebSocket listener becomes available: the 60-second maximum retry delay plus a 5-second handshake/test margin. Test the retry schedule separately. | Phase 8 |
